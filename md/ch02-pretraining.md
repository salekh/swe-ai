# Chapter 2: LLM Pre-Training — From Raw Text to World Models

On day 47 of a 65-day training run, the loss curve spiked. The training cluster—4,096 H100 GPUs at $2/GPU-hour—was burning $196,000 per day. The team had 20 minutes to decide: roll back to a 12-hour-old checkpoint (losing $98,000 of compute), or hope the spike was transient.

It didn't recover. They rolled back, diagnosed the cause (a corrupted data shard containing a 2GB auto-generated file of the same line repeated 50 million times), fixed the pipeline, and resumed. Total cost: ~$150,000 and two days. Nobody wrote a postmortem. This is just Tuesday in pre-training.

## The Training Data Pipeline

*[Figure: Training data pipeline: collection → cleaning → deduplication → tokenization]*

A language model is only as good as its training data. The data pipeline is a massive-scale ETL system processing petabytes.

### Data Collection

Multiple sources:
- **Web crawls**: Common Crawl (billions of pages)
- **Books**: Digitized from Internet Archive, publisher partnerships
- **Code**: GitHub repositories, filtered by license and quality
- **Scientific literature**: ArXiv, PubMed, Semantic Scholar
- **Wikipedia**: High-quality encyclopedic content
- **Curated datasets**: StackOverflow, Reddit, specialized forums
- **Synthetic data**: Increasingly, data generated by existing models

LLaMA proportions: ~67% web, 15% code, 4.5% Wikipedia, 4.5% books, 2.5% ArXiv, 6.5% other.

### Data Cleaning and Filtering

Production pipeline stages:
1. **Language identification**: fastText classifier, remove non-target languages
2. **Quality filtering**: Perplexity, length, special character ratio, repetition filtering
3. **Content filtering**: Remove toxic, biased, or PII content
4. **Deduplication**: Exact-match (hash-based) + fuzzy (MinHash/LSH)

### Deduplication

The most impactful cleaning step. Web data contains enormous amounts of duplication. Deduplicated training data consistently produces better models—duplicated content causes the model to memorize rather than generalize.

Two approaches:
- **Exact deduplication**: Hash each document, remove duplicates. Fast, catches copy-paste.
- **Fuzzy deduplication**: MinHash + LSH to find near-duplicates. Catches slight variations (boilerplate text, templates).

## Tokenization

### Why Not Just Use Characters or Words?

- Characters: vocabulary is small (~256), but sequences are very long and expensive
- Words: vocabulary is huge (500K+), many rare/unseen words
- **Subword tokenization**: The sweet spot—~32K–128K vocabulary, handles rare words by decomposition

### Byte Pair Encoding (BPE)

The dominant tokenization algorithm. Iteratively merges the most frequent adjacent pairs of tokens until the vocabulary reaches the desired size. `"unhappiness"` → `["un", "happiness"]` or `["un", "happ", "iness"]`.

### Tokenizer Design Trade-offs

- **Vocabulary size**: Larger = shorter sequences but bigger embedding table
- **Character coverage**: Must handle all languages and special characters
- **Code handling**: Must tokenize programming languages efficiently
- **Consistency**: Tokens should map to meaningful linguistic units

> ⚠️ **Tokenization is a one-way door**: Once you've trained a model with a specific tokenizer, you cannot change it without retraining. Choose carefully.

## The Transformer Architecture: A Systems Perspective

*[Figure: Transformer architecture with data flow]*

### The Forward Pass as a Data Pipeline

Think of the Transformer as a multi-stage data pipeline:

1. **Token embedding**: Map each token to a dense vector (lookup table)
2. **Position encoding**: Add positional information (RoPE in modern models)
3. **Repeated transformer blocks** (N layers, typically 32–128):
   - Layer normalization
   - Multi-head self-attention (the core operation)
   - Feedforward network (2-layer MLP with GeLU activation)
   - Residual connections (skip connections for gradient flow)
4. **Output projection**: Map final hidden state to vocabulary logits
5. **Softmax**: Convert logits to probability distribution over next token

### Attention as a Database Query

Self-attention is conceptually identical to a database join:
- **Query (Q)**: "What am I looking for?"
- **Key (K)**: "What do I contain?"
- **Value (V)**: "What information do I have?"

The attention score `softmax(QK^T / √d_k) × V` computes a weighted average of all values, where the weights come from how well each key matches the query. Complexity: O(n²) in sequence length.

### Multi-Head Attention: Parallel Queries

Multi-head attention runs multiple attention operations in parallel, each with different learned projections. Different heads learn to attend to different types of relationships (syntactic, semantic, positional).

### Scaling Laws: Bigger is (Usually) Better

Kaplan et al. (2020) and Hoffmann et al. (2022, "Chinchilla") established empirical scaling laws:
- Performance scales as a power law with model size, dataset size, and compute
- **Chinchilla optimal**: For a given compute budget, there's an optimal ratio of model size to training data (~20 tokens per parameter)
- Implication: A 70B model needs ~1.4 trillion training tokens

## Pre-Training Infrastructure at Scale

### Distributed Training Strategies

*[Figure: 3D Parallelism — Data × Tensor × Pipeline]*

Three dimensions of parallelism:

- **Data Parallelism**: Each GPU gets a copy of the model and processes different data batches. Gradients are synchronized via all-reduce. Simple but requires the model to fit on one GPU.
- **Tensor Parallelism**: Split individual layers across GPUs. The attention computation is partitioned so each GPU computes a subset of heads. Requires fast interconnect (NVLink).
- **Pipeline Parallelism**: Split the model layer-wise across GPUs. GPU 1 handles layers 1–16, GPU 2 handles layers 17–32. Introduces pipeline bubbles (idle time).

Production training uses all three simultaneously ("3D parallelism").

### Hardware: The GPU Cluster

A frontier training run requires:
- **4,096–16,384 GPUs** (H100 or B200)
- **InfiniBand or NVLink** interconnect (400–800 Gbps per GPU)
- **Petabytes of storage** (parallel filesystem, often Lustre or GPFS)
- **Redundant power and cooling** for sustained multi-month runs

### Checkpointing and Fault Tolerance

At 10,000 GPUs, hardware failures happen daily. Production training systems:
- Checkpoint every 15–30 minutes (save full model state + optimizer state)
- Each checkpoint: 2–10 TB of data (model weights + optimizer state + RNG state)
- Automatic failure detection and recovery
- Elasticity: continue training with fewer GPUs when failures occur

## Training Dynamics and Stability

### Loss Curves and Monitoring

The training loss curve is the primary diagnostic tool. Healthy training shows a smooth, monotonically decreasing curve. Warning signs:
- **Loss spikes**: Usually caused by data quality issues or learning rate problems
- **Loss plateaus**: Model has extracted all information at current scale
- **Divergence**: Loss increases and doesn't recover—training has failed

### Learning Rate Schedule

Most modern training runs use a warmup + cosine decay schedule:
1. **Warmup** (first 1–5% of training): Linear increase from 0 to peak learning rate
2. **Cosine decay**: Gradual decrease following a cosine curve to ~10% of peak

### Mixed-Precision Training

Modern training uses mixed precision (bf16/fp32) to:
- Halve memory requirements per parameter
- Double throughput on tensor cores
- Maintain numerical stability with fp32 master weights and loss scaling

## Inference: The KV-Cache and Why It Matters

*[Figure: KV-Cache operation during autoregressive generation]*

During generation, the model produces tokens one at a time. Without caching, generating token N requires recomputing attention over all N-1 previous tokens—O(n²) work for each new token.

The **KV-Cache** stores the Key and Value tensors from all previous tokens. When generating token N+1, only the new token's Q, K, V need to be computed. The cached K and V from tokens 1–N are reused. This reduces per-token cost from O(n²) to O(n).

**Memory cost**: For a 70B model generating 4096 tokens, the KV-cache requires ~40GB of GPU memory—often more than the model weights themselves. This is why long-context generation is expensive.

Optimizations:
- **Grouped Query Attention (GQA)**: Share KV heads across multiple query heads (4–8× cache reduction)
- **Multi-Query Attention (MQA)**: Single KV head shared across all query heads
- **Paged KV-Cache** (vLLM): Manage KV-cache memory like virtual memory pages for efficient batching

## Notable Pre-Training Runs

### GPT-4 and the Mixture of Experts Architecture

GPT-4 (widely reported) uses a Mixture of Experts (MoE) architecture: multiple "expert" sub-networks with a routing layer that activates only a subset for each token. This allows scaling total parameters without proportionally increasing compute per forward pass.

### Gemini: Multimodal Pre-Training

Google's Gemini models are natively multimodal—trained on text, images, video, and audio from the start, rather than bolting on vision after text pre-training. This produces better cross-modal understanding.

### LLaMA: The Open-Source Catalyst

Meta's LLaMA models demonstrated that open-source models with careful data curation could approach proprietary model quality at a fraction of the cost, catalyzing the open-source LLM ecosystem.

## Chapter Summary

1. **Pre-training is a systems engineering problem**: The ML is (relatively) straightforward; the hard part is managing petabyte-scale data pipelines, coordinating thousands of GPUs, and recovering from hardware failures
2. **Data quality > model size**: A smaller model trained on cleaner, better-curated data outperforms a larger model on noisy data
3. **Scaling laws are empirical guides**: Chinchilla-optimal training balances model size and training data (~20 tokens per parameter)
4. **Distributed training = 3D parallelism**: Data, tensor, and pipeline parallelism combined
5. **Inference is memory-bound**: The KV-cache determines serving cost; optimizations like GQA are essential for production
6. **Tokenization is a one-way door**: Choose carefully; it cannot be changed without retraining
